Create `scripts/soak-test.ts` -- a standalone Node.js script (run via `npx tsx scripts/soak-test.ts`) that simulates Claude Code sessions for a configurable duration.

**Purpose:** Drive the app with realistic session churn so memory/resource behavior can be observed during the 8-hour soak test. The user starts Agent World normally, then runs this script in a separate terminal.

**CLI interface:**
```
npx tsx scripts/soak-test.ts [--dir PATH] [--duration HOURS] [--sessions N]
  --dir       Override Claude projects dir (default: ~/.claude/projects)
  --duration  How many hours to run (default: 8)
  --sessions  Max concurrent simulated sessions (default: 4)
```

**Simulation behavior:**
1. Create a temp subdirectory under `--dir` named `soak-test-TIMESTAMP` (so it doesn't collide with real projects)
2. Maintain a pool of 1 to `--sessions` simulated sessions
3. On a random interval (every 5-15 seconds), perform ONE of these actions:
   - **Add session:** Create a new UUID.jsonl file with an initial `{"type":"user","message":"soak test session"}` line (if below max sessions)
   - **Update session:** Append a new JSONL line to an existing session with a random type from `["assistant", "user", "progress"]` and random tool_name from `["Edit", "Read", "Bash", "Write", "Grep"]`, updating its mtime (triggers status change in the app)
   - **Remove session:** Delete a random JSONL file (triggers fade-out in the app)
4. Each JSONL line must be valid JSON on a single line, matching the format the session detector parses: `{"type":"assistant","message":{"usage":{"input_tokens":100,"output_tokens":50}}}`  -- but usage fields are optional, only type is required for status detection
5. Log to stdout every action taken: `[soak] +session UUID | [soak] ~update UUID (assistant/Edit) | [soak] -session UUID`
6. Log a periodic heartbeat every 60 seconds: `[soak] heartbeat: N active sessions, M total actions, elapsed Xh Ym`
7. On SIGINT (Ctrl+C), clean up all created JSONL files and the temp directory, log summary, and exit

**Structure for testability:**
- Export `generateJsonlEntry(type: string, toolName?: string): string` as a pure function
- Export `generateSessionId(): string` as a pure function
- Export `parseArgs(argv: string[]): { dir: string; duration: number; sessions: number }` as a pure function
- Keep the main simulation loop as the default export or in a `main()` that runs only when script is executed directly (`if (require.main === module)` pattern -- but since using ESM via tsx, use `import.meta.url` check or just put main logic after exports)

**JSONL line format for different entry types:**
```json
{"type":"user","message":"soak test prompt","timestamp":"2026-03-16T20:00:00.000Z"}
{"type":"assistant","message":{"content":[{"type":"text","text":"response"}],"usage":{"input_tokens":100,"output_tokens":50,"cache_creation_input_tokens":0,"cache_read_input_tokens":0}},"timestamp":"2026-03-16T20:00:05.000Z"}
{"type":"progress","tool_name":"Edit","timestamp":"2026-03-16T20:00:03.000Z"}
```

**Important:** Use only Node.js built-in modules (fs, path, os, crypto, process). No npm dependencies needed.

Create `scripts/analyze-soak.ts` -- a standalone Node.js script that reads crash.log, extracts memory stats, and generates a structured pass/fail report for the STAB-03 soak test criteria.

**CLI interface:**
```
npx tsx scripts/analyze-soak.ts [--log PATH] [--threshold MB]
  --log        Path to crash.log (default: %APPDATA%/agent-world/crash.log on Windows)
  --threshold  Max allowed growth in MB (default: 50)
```

**Analysis performed:**

1. **Parse [MEMORY] stats lines:**
   Extract all lines matching `[MEMORY] heap=XX.XMB rss=YY.YMB` -- capture timestamp, heapMB, rssMB for each sample.

2. **Memory growth analysis:**
   - First sample heap, last sample heap, delta = last - first
   - Min and max heap values across entire run
   - PASS if delta < threshold (default 50MB), FAIL otherwise
   - Calculate run duration from first sample timestamp to last sample timestamp

3. **Monotonic trend detection:**
   - For each metric (heap, rss): check if values are monotonically non-decreasing
   - Define "monotonic" as: 80%+ of consecutive sample pairs show growth (allows for GC dips)
   - Also report the longest consecutive increasing run
   - PASS if no metric is monotonically increasing, FAIL otherwise

4. **Crash/critical detection:**
   - Count lines containing `[CRASH]` or `[CRITICAL]`
   - PASS if zero crash/critical entries, FAIL if any

5. **Memory warning detection:**
   - Count lines matching `[warn] [MEMORY]` (the trend warnings from MemoryMonitor)
   - Report count (informational, not a pass/fail criterion by itself)

**Report output format (stdout):**
```
=== SOAK TEST REPORT ===

Duration: 8h 02m (482 samples at 60s intervals)

--- Memory Growth ---
First heap: 42.3 MB
Last heap:  55.1 MB
Growth:     12.8 MB (threshold: 50 MB)
Min heap:   41.0 MB | Max heap: 58.2 MB
Result:     PASS

--- Monotonic Trend ---
Heap:  21% increasing pairs (longest run: 8) -- PASS
RSS:   18% increasing pairs (longest run: 5) -- PASS
Result: PASS

--- Crash Events ---
[CRASH] entries:    0
[CRITICAL] entries: 0
[ERROR] entries:    3 (non-blocking)
Result: PASS

--- Memory Warnings ---
Trend warnings: 0

=== OVERALL: PASS ===
(or)
=== OVERALL: FAIL ===
Failed criteria:
  - Memory growth 62.3 MB exceeds 50 MB threshold
```

Exit code: 0 for PASS, 1 for FAIL.

**Structure for testability:**
- Export `parseMemoryLine(line: string): { timestamp: string; heapMB: number; rssMB: number } | null`
- Export `analyzeGrowth(samples: { heapMB: number }[], thresholdMB: number): { firstHeap: number; lastHeap: number; delta: number; min: number; max: number; pass: boolean }`
- Export `detectMonotonicTrend(values: number[]): { isMonotonic: boolean; increasingPct: number; longestRun: number }`
- Export `countCrashEntries(lines: string[]): { crashes: number; criticals: number; errors: number }`
- Export `parseArgs(argv: string[]): { logPath: string; thresholdMB: number }`

**Important:** Use only Node.js built-in modules (fs, path, os, readline, process). No npm dependencies.
