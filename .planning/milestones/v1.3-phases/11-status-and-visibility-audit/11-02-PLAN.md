```typescript
private advanceStatusDebounce(sessionId: string, deltaMs: number): SessionStatus | null {
  const debounce = this.statusDebounce.get(sessionId);
  if (!debounce) return null;
  const rawStatus = this.lastRawStatus.get(sessionId);
  if (!rawStatus) return debounce.committedStatus;
  if (rawStatus === debounce.committedStatus) {
    debounce.pendingStatus = rawStatus;
    debounce.timer = 0;
    return null;
  }
  if (rawStatus === debounce.pendingStatus) {
    debounce.timer += deltaMs;
    if (debounce.timer >= STATUS_DEBOUNCE_MS) {
      debounce.committedStatus = rawStatus;
      debounce.timer = 0;
      this.lastCommittedStatus.set(sessionId, rawStatus);
      return rawStatus;
    }
  } else {
    debounce.pendingStatus = rawStatus;
    debounce.timer = 0;
  }
  return null;
}
```

**Fix 1: `cancelFadeOut()` -- reset all visual state, not just animation**

The current `cancelFadeOut()` resets state to `idle_at_hq`, alpha to 1, and animation to idle. But it does NOT reset visual status properties that may have accumulated during the fade. Specifically, `isBreathing` might be true from a previous waiting status, and `tint` might be stale.

Replace the current `cancelFadeOut()` method with:
```typescript
/**
 * Cancel an in-progress fade-out and return to idle_at_hq state.
 * Used when an idle-timeout fade is interrupted by session reactivation.
 * Fully resets all visual state to prevent stale effects from lingering.
 * No-op if agent is not currently fading out.
 */
cancelFadeOut(): void {
  if (this.state !== 'fading_out') return;
  this.state = 'idle_at_hq';
  this.fadeOutTimer = 0;
  this.alpha = 1;
  // Reset breathing in case it was active before fade started
  this.isBreathing = false;
  this.breathTimer = 0;
  // Reset tint to default (active) since the agent is being reactivated
  this.currentTint = 0xffffff;
  this.targetTint = 0xffffff;
  this.tintTimer = 0;
  this.tint = 0xffffff;
  this.setAnimation('idle');
}
```

**Why:** When a fading agent is cancelled (user starts a new task in a session that was idle), the agent returns to `idle_at_hq` but may retain breathing/tint state from its previous visual status. Since `applyStatusVisuals()` is a no-op during fade-out, the visual status hasn't been updated. By resetting to clean defaults, the next `applyStatusVisuals()` call (from World's debounce) will apply the correct state cleanly.

**Fix 2: Ensure `applyStatusVisuals()` always forces alpha=1 on non-breathing transitions**

The current code already does `this.alpha = 1` when `!this.isBreathing`. This is correct, but there's a subtle issue: if the breathing timer was at a low point (alpha 0.5) and a rapid status change from waiting->active->waiting happens within the debounce window, the alpha could be stuck at whatever the breathing cycle left it at. The current code handles this correctly because `applyStatusVisuals()` is only called when the debounce commits, and at that point `isBreathing` is set based on the committed status.

However, add a safety assertion. After the line `this.isBreathing = (status === 'waiting');`, add a comment and explicit alpha reset for the non-breathing case that also resets `breathTimer`:

Change:
```typescript
// Breathing: only when waiting
this.isBreathing = (status === 'waiting');
if (!this.isBreathing) {
  this.alpha = 1; // Reset alpha when not breathing
}
```

To:
```typescript
// Breathing: only when waiting
this.isBreathing = (status === 'waiting');
if (!this.isBreathing) {
  this.alpha = 1; // Force alpha=1 to clear any breathing oscillation residue
  this.breathTimer = 0; // Reset timer so next waiting period starts from consistent state
}
```

**Why:** Without resetting `breathTimer`, a session that goes active->waiting->active->waiting could resume breathing at a random phase, potentially starting at a low alpha value. Resetting the timer ensures each waiting period starts its breathing cycle consistently.

**Fix 1: Route agent to correct building after cancelFadeOut, not just idle_at_hq**

In `manageAgents()`, after `agent.cancelFadeOut()` is called (around line 405-406), the code currently falls through to the routing logic below. However, the agent is now in `idle_at_hq` state (set by cancelFadeOut). The routing logic will then check `agentState` which will be `idle_at_hq`, and if `activityType !== 'idle'`, it will correctly send the agent to a building. This is actually correct behavior already.

BUT there's a bug: after cancelFadeOut, the agent's debounce state is stale. The `statusDebounce` map still has the old committed status (likely `idle`), and the `lastRawStatus` still has whatever the fade was triggered by. We need to reinitialize the debounce for the reactivated agent so the new status is applied cleanly.

After the `agent.cancelFadeOut();` line, add debounce reinitialization:

Replace:
```typescript
if (agent.getState() === 'fading_out') {
  if (session.activityType !== 'idle' || session.status !== 'idle') {
    agent.cancelFadeOut();
    // Fall through to routing with fresh state

  }

```

With:
```typescript
if (agent.getState() === 'fading_out') {
  if (session.activityType !== 'idle' || session.status !== 'idle') {
    agent.cancelFadeOut();
    // Reinitialize debounce state for clean reactivation
    this.statusDebounce.set(session.sessionId, {
      pendingStatus: session.status,
      timer: 0,
      committedStatus: session.status,
    });
    this.lastCommittedStatus.set(session.sessionId, session.status);
    this.lastRawStatus.set(session.sessionId, session.status);
    // Apply visual status immediately (bypass debounce for reactivation)
    agent.applyStatusVisuals(session.status);
    // Fall through to routing with fresh state
  } else {
    continue; // Still idle, let fade-out proceed
  }
}
```

**Why:** Without reinitializing debounce, the agent's visual status would remain stale (whatever it was before fade-out began) until the debounce timer naturally commits. This could take 2.5 seconds, during which the agent appears with incorrect tint/breathing. Reinitializing gives the agent immediate correct visuals upon reactivation.
