**Fix 1: `determineStatus()` -- system entries should map to `waiting` not `idle`**

In the `determineStatus()` method of `FilesystemSessionDetector`, change the `'system'` case. Currently:
```typescript
case 'system':
  // System entries (e.g., turn_duration) indicate task boundaries
  return timeSinceModified < 5000 ? 'active' : 'idle';
```

Change to:
```typescript
case 'system':
  // System entries (e.g., turn_duration) mark task completion boundaries.
  // After completion, the session is waiting for user input, not idle.
  // Only report idle after the full IDLE_THRESHOLD_MS has passed (handled above).
  return timeSinceModified < 5000 ? 'active' : 'waiting';
```

**Why:** When Claude finishes a task, it writes a `system` entry (type: `turn_duration`). After this, the session is waiting for the user to respond. The previous code reported `idle` after 5 seconds, but the session is actually in `waiting` state until the 30-second IDLE_THRESHOLD_MS kicks in. With the fix, the flow is: system entry written -> 0-5s: `active` (just completed) -> 5-30s: `waiting` (awaiting user input) -> 30s+: `idle` (via the timeSinceModified > IDLE_THRESHOLD_MS guard at the top of the method).

**Fix 2: Stale session filter -- preserve sessions with recent cached status of `waiting` or `active`**

In the `processSessionFile()` method, the stale session check currently unconditionally returns null for files not modified in 30+ minutes:
```typescript
if (now - stat.mtimeMs > STALE_SESSION_MS) {
  return null;
}
```

Replace this with a check that preserves sessions whose last known status was `waiting` (user hasn't responded yet):
```typescript
if (now - stat.mtimeMs > STALE_SESSION_MS) {
  // Preserve sessions that were actively waiting -- user may not have responded yet.
  // Only filter truly stale sessions (last known status was idle or unknown).
  const cached = this.mtimeCache.get(sessionId);
  if (cached && (cached.sessionInfo.status === 'waiting' || cached.sessionInfo.status === 'active')) {
    // Session was waiting/active when last checked -- keep it visible.
    // Update status to waiting (file hasn't changed, so it's definitely waiting now).
    const preserved: SessionInfo = { ...cached.sessionInfo, status: 'waiting' };
    return preserved;
  }
  return null;
}
```

**Why:** A Claude session can legitimately wait for user input for hours (user went to lunch, got distracted). The session file stops being modified, but the session is still alive and waiting. Filtering it out removes the agent from the world, making the user think the session ended. This fix keeps the agent visible with `waiting` status as long as the last known status was not `idle`.

**Fix 3: Fix the stale check comment**

The comment above the stale check says "not modified in over 1 hour" but the actual threshold is 30 minutes. Fix the comment to match reality:

This comment fix applies to the code BEFORE the replacement in Fix 2. After Fix 2, the comment becomes part of the new block, so just ensure the new code's comments are accurate.
